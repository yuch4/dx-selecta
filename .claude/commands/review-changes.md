# Review Changes

直近の実装コードをレビューする。
**Read-onlyモード**で修正は行わず、問題点と改善提案のみを報告。

## 入力
$1 = レビュー対象（例: 直近のcommit / 特定ファイル）
$2 = フォーカス領域（任意、例: セキュリティ / パフォーマンス）

## 参照
- 開発規約: `docs/development-guidelines.md`
- アーキテクチャ: `docs/architecture.md`

## レビュー手順

### 1) 対象の特定
```bash
# 直近のコミット差分
git diff HEAD~1

# 直近N件のコミット一覧
git log --oneline -10
```

### 2) レビュー観点

#### 🔒 セキュリティ
- [ ] 認証・認可の適切な実装
- [ ] 入力値のバリデーション
- [ ] SQLインジェクション/XSS対策
- [ ] シークレット・機密情報の露出

#### 🏗️ アーキテクチャ・パターン準拠
- [ ] 既存の命名規則に従っている
- [ ] ファイル配置が規約通り
- [ ] 責務分離

#### 📝 コード品質
- [ ] 型安全性
- [ ] エラーハンドリングの網羅性
- [ ] 重複コードの有無

#### ⚡ パフォーマンス
- [ ] 不要な再レンダリング
- [ ] N+1クエリの可能性

#### 🧪 テスタビリティ
- [ ] テストが追加/更新されている

## 出力フォーマット

```markdown
## 📋 レビューサマリー

**対象**: [レビュー対象の概要]
**変更ファイル数**: X files

### 🎯 総合評価

| 観点 | 評価 | コメント |
|------|------|----------|
| セキュリティ | ✅/⚠️/❌ | 一言 |
| パターン準拠 | ✅/⚠️/❌ | 一言 |
| コード品質 | ✅/⚠️/❌ | 一言 |

### 🚨 Critical（必須対応）
1. **[ファイル名:行番号]** 問題の説明
   - 提案: 修正案

### ⚠️ Warning（推奨対応）
1. **[ファイル名:行番号]** 問題の説明

### 💡 Suggestion（任意改善）
1. 提案内容

### ✅ Good Points
1. 良かった点
```
