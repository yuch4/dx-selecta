# PRD Writing Guide

PRD（プロダクト要求定義書）作成の詳細ガイド。

## 良いPRDの条件

| 観点 | 良い例 | 悪い例 |
|------|--------|--------|
| ビジョン | 「選定から稟議までの時間を1/3に短縮」 | 「便利なツールを作る」 |
| ペルソナ | 「35歳、情シス兼総務、年2-3件のSaaS導入」 | 「ビジネスユーザー」 |
| KPI | 「DAU 100人（3ヶ月後、Auth ログイン数で測定）」 | 「ユーザーを増やす」 |
| 受入条件 | 「入力完了から結果表示まで3秒以内」 | 「高速に動作する」 |
| 非機能 | 「同時100ユーザー、レスポンス500ms以内」 | 「スケーラブル」 |

## セクション別ガイド

### プロダクトビジョン
- **3-5文**で簡潔に
- 「誰の → どんな課題を → どう解決し → どんな状態になるか」の構造
- 定量的な効果があれば含める

### ペルソナ
- **具体的な人物像**（名前、年齢、職業）
- **現在の課題**を3つ以上
- **1日のワークフロー**で「いつ・何に困っているか」を可視化

### KPI（SMART形式）
| 要素 | 説明 | 例 |
|------|------|-----|
| Specific | 具体的 | 「DAU」ではなく「ログイン済みユーザー数」 |
| Measurable | 測定可能 | 測定方法を明記 |
| Achievable | 達成可能 | 現実的な数値 |
| Relevant | 関連性 | ビジネス目標との紐付け |
| Time-bound | 期限 | 「3ヶ月後」など |

### ユーザーストーリー
```
[ペルソナ]として、[目的]のために、[機能]が欲しい
```

### 受入条件
- **測定可能**な形で書く
- **正常系**だけでなく**エッジケース**も含める
- チェックボックス形式で完了判定しやすく

### 非機能要件
- **数値で定義**（「速い」ではなく「500ms以内」）
- **測定方法**を明記
- カテゴリ：パフォーマンス / 信頼性 / セキュリティ / ユーザビリティ

## 品質チェックリスト

### ビジョンと目標
- [ ] ビジョンが3-5文で明確に書かれている
- [ ] 「誰の・何を・どう解決」が読み取れる
- [ ] 定量的な効果（あれば）が含まれている

### ターゲットユーザー
- [ ] ペルソナが具体的（名前・年齢・職業）
- [ ] 課題が3つ以上列挙されている
- [ ] 1日のワークフローが書かれている

### 成功指標（KPI）
- [ ] SMART形式で書かれている
- [ ] 測定方法が明記されている
- [ ] 期限がある

### 機能要件
- [ ] ユーザーストーリー形式で書かれている
- [ ] 受入条件が測定可能
- [ ] 優先度（P0/P1/P2）がある

### 非機能要件
- [ ] パフォーマンスが数値で定義されている
- [ ] セキュリティ要件が明記されている

### スコープ
- [ ] スコープ外が明示されている
- [ ] MVP機能とPost-MVP機能が区別されている

## よくある問題と対策

| 問題 | 原因 | 対策 |
|------|------|------|
| ビジョンが曖昧 | 「何を解決するか」が不明確 | 課題→解決→効果の構造で書き直す |
| ペルソナが抽象的 | 「ユーザー」で止まっている | 具体的な人物像とワークフローを追加 |
| KPIが測定不能 | 測定方法を考えていない | 測定手段を先に決める |
| 受入条件が曖昧 | 「〇〇できる」で止まっている | 数値・条件を追加 |
